# FN 루프 — React 프론트엔드(KRDS/전자정부 스타일) 구축 루프

> 용도: **코인 자동매매 웹의 프론트엔드(React·차트/대시보드·UI/UX·반응형)**만 담당하는 루프. BN 루프와 **다른 REPO·다른 세션**에서 동시 실행.
> 저장소: FN = COIN_FN / BN = COIN_BN (별개 repo). 둘은 **비동기 REST(요청-응답) + WebSocket**으로만 연결.
> 근거 문서: [UI_GUIDE.md](UI_GUIDE.md)(UI 개선 플레이북·백로그·KRDS/KWCAG).
> 사용법: 아래 코드블록을 통째로 복사 → `/loop 10m <붙여넣기>`.
> 매 사이클 **작업 상태를 `FN.md`에 보고**하고, **무엇이 업데이트됐는지 `작업완료보고.md`에도 기록**한 뒤 **매 사이클 COIN_FN GitHub에 commit+push**한다.

```
[FN 루프 — 코인 자동매매 React 프론트엔드(KRDS 스타일) 한 사이클]

역할: 코인 자동매매 웹(업비트, 차트 실시간 관찰 + 자동/수동 매매 컨트롤)의 프론트엔드를 React로 구축·개선한다. "보이는 부분"만 담당. BN(COIN_BN)과는 다른 repo이며 비동기 REST/WS로만 통신한다.

★ 크로스-repo 연결 원칙 (매 사이클 최우선):
- FN과 BN은 별개 repo. FN은 BN의 소스가 아니라 "API 계약"만 소비한다.
- 매 사이클 시작 시 BN의 API 문서를 확인해 UI를 동기화한다:
  · 1순위: 실행 중인 BN의 OpenAPI 스펙(FastAPI 자동 생성) — http://localhost:8000/openapi.json 및 Swagger UI /docs.
  · 대안: BN repo가 공개한 API 계약 문서(예: raw GitHub의 API.md/openapi.json) 또는 BN.md의 "FN 계약" 항목.
- BN 스펙을 현재 UI가 지원하는 엔드포인트 목록과 대조(diff) → 새로 생기거나 바뀐 엔드포인트를 찾는다.
- 새 BN 기능이 있으면 그에 맞춰 **필요하면 새 페이지를 생성하고, 그 기능에 맞는 버튼/폼/표를 만든다**(예: 새 전략 토글, 새 주문타입, 백테스트 파라미터 입력, 봇 설정). UI가 BN 기능을 항상 따라잡게 유지.
- 모든 데이터 요청은 비동기(fetch/axios, async/await)로, 로딩/성공/빈상태/에러 4상태를 반드시 처리. CORS·타임아웃·재시도 고려.

스택: React + Vite(TS 권장) · 상태 훅/Context(과설계 금지) · 데이터패칭 fetch/axios(+ TanStack Query 선택) · 차트 lightweight-charts(UI_GUIDE §3) · 실시간 WebSocket.

★ 디자인 원칙(최우선) — "관공서 업무화면처럼 정직·기능적" (UI_GUIDE §1·2):
- 한국 정부 표준 KRDS(옛 전자정부 UI가이드 승계) 준수: 절제된 색, 명확한 정보 위계, 표준 폼, 표 중심, KWCAG 접근성(대비 4.5:1·키보드·aria·label).
- "AI스러운 것" 전면 금지: 그라디언트·글래스모피즘·네온/글로우·보라·과한 그림자·둥근 말풍선·이모지·일러스트·화려한 히어로 금지.
- 깔끔·심플: 흰 배경 + 무채색 + 절제된 포인트색. 평면(flat)·1px 실선·여백 구분.
- border-radius 0 지향(--r:0~2px). 둥근 버튼/카드/입력창 금지. 각진 모서리 기본.
- ★ 상승/하락색: 한국식 상승=빨강·하락=파랑 + 색맹 대응 부호(+/−)·화살표 병기.
- 숫자 가독성(등폭 숫자). HTS/관제화면 톤.

범위(내 담당): React 컴포넌트/훅/스타일/라우팅/데이터패칭/차트/표/폼/레이아웃/반응형. 주요 화면(UI_GUIDE §5 백로그): 실시간 차트+지표 오버레이·신호 마커 / 포지션·자산 패널 / 컨트롤 바(자동매매 ON·OFF·수동주문·모드 배지·킬스위치) / 거래로그·백테스트 요약 / 봇 상태.
금지(BN 담당): FastAPI·신호 계산·주문 실행·거래소 API·백테스트 산식. 신호/주문/계산은 프론트 재구현 금지 — BN API 호출·결과 표시·상태만.

이번 사이클 산출물 — 하나를 명확히:
A) BN 동기화(신규): BN API 스펙 diff에서 발견한 새 기능에 맞춰 페이지/버튼/폼/표 생성·연결.
B) 구축/개선: UI_GUIDE 백로그의 한 화면 조각을 구현하거나 정보위계·스타일·접근성 개선.
C) 통합/반응형: 비동기 연동(로딩/에러/빈상태) + 실시간 WS 반영 + mobile(375)/tablet(768) 사용성.
→ 매 사이클 A(동기화 diff)를 먼저 점검. 새 BN 기능 있으면 A 우선, 없으면 B/C.

매 사이클 순서:
1. FN.md 최신 항목 + UI_GUIDE.md 읽어 다음 대상 파악.
2. ★ BN API 스펙 확인: /openapi.json(또는 공개 계약)와 현재 UI 지원 목록 diff → 신규/변경 엔드포인트 목록화.
3. 서버 확인: React dev(Vite) + BN 기동(또는 mock). BN 미가동이면 스펙 기반 mock으로 진행하고 FN.md에 "BN 실연동 미검증" 명기.
4. 변경 전 git pull(COIN_FN).
5. 산출물 A/B/C 선정(이유 한 줄) → 구현. 비동기 요청은 로딩/성공/빈/에러 4상태 처리.
6. 프리뷰 검증: mobile(375)·tablet(768)·desktop(1280) + 다크/라이트. 렌더/차트/마커/테마 깨짐 0. 금지스타일 잔재(radius·그림자·그라디언트) 0. 대비 4.5:1. 상승/하락 색+부호. 콘솔 에러 0.
7. FN.md 맨 위 새 항목(형식 아래).
8. ★ 작업완료보고.md 갱신: 이번 사이클에 무엇이 업데이트됐는지(생성/변경 화면·엔드포인트 연동·수정 내용)를 맨 위에 요약 추가.
9. ★ 매 사이클 반드시 COIN_FN GitHub에 commit + push (FN.md·작업완료보고.md·소스 포함). 접두사 "[FN-사이클 N]". 루프 종료가 아니라 매 사이클마다 원격 반영.

FN.md 보고 형식:
## [FN-사이클 N] YYYY-MM-DD HH:MM — 제목
- 산출물: A(BN동기화) | B(구축/개선) | C(통합/반응형)
- BN 스펙 diff: 이번에 반영한 신규/변경 엔드포인트(없으면 "없음")
- 변경: 무엇을 왜 (대상 화면·생성한 페이지/버튼)
- 검증: mobile/tablet/desktop·다크/라이트(스샷/DOM)·금지스타일 0·대비·콘솔·BN 실연동/미검증
- BN 요청: 필요한 API 계약(부족한 엔드포인트/필드)
- 다음 후보: …
- 커밋: hash / 메시지

작업완료보고.md 형식(맨 위에 누적 추가):
## [FN-사이클 N] YYYY-MM-DD HH:MM
- 업데이트: 이번 사이클에 무엇이 바뀌었나(생성한 페이지/버튼·연동한 BN 엔드포인트·스타일/접근성 수정)
- BN 연동: 반영한 신규/변경 API(없으면 "없음")
- 커밋: hash

[상시 UI/UX 기준 — 위반 시 즉시 수정] (UI_GUIDE §6 체크리스트)
- KRDS/KWCAG: 절제된 색·표 중심·표준 폼·label/aria·대비 4.5:1.
- border-radius 0 지향(--r:0~2px). 둥근모서리·글래스·그라디언트·네온·그림자·이모지·일러스트 금지.
- 라이트 기본 + 다크 토글(localStorage). 색·여백·테두리는 CSS 변수(토큰).
- 상승=빨강/하락=파랑 + 부호(+/−) 병기. 신호/주문/계산은 BN API 결과 표시만.
- "예뻐 보이려는" 장식 변경은 개선 아님 — 읽는속도·오해방지·안전·접근성·반응형·성능만 개선으로 인정.

[하드 규칙]
- FN·BN은 별개 repo, 비동기 REST/WS로만 결합. FN에서 BN 소스/로직 변경 금지.
- 실제 주문·이체를 프론트에서 직접 호출 금지 — 반드시 BN 엔드포인트 경유. 위험 동작(전체청산·live 전환)은 확인 다이얼로그 필수.
- API 키/시크릿 프론트 노출 절대 금지(.env·번들 포함 금지). node_modules·dist 커밋 금지(.gitignore 확인).
- 검증 못 한 부분(BN 실연동 등)은 FN.md에 "미검증" 명기.
```
